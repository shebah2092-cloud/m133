## 1. ما هو acados؟

### التعريف البسيط
acados هو **برنامج يحسب أفضل قرار تحكم ممكن في كل لحظة**. بدلاً من استخدام قواعد ثابتة مثل PID (إذا كان الخطأ كذا، اعمل كذا)، يقوم acados بحل مسألة رياضية كاملة في كل دورة تحكم ليجد **أفضل أمر تحكم** يحقق الهدف مع احترام جميع القيود.

### التشبيه
- **PID** مثل سائق يتبع تعليمات بسيطة: "إذا انحرفت يمينًا، أدر المقود يسارًا"
- **acados (MPC)** مثل سائق يرى الطريق أمامه ويخطط أفضل مسار مع مراعاة حدود السرعة والمنعطفات والوقود

### الاسم الرسمي
acados = **A**cademic **C**ode for **A**utomation of **D**ynamic **O**ptimization **S**olvers

### الخصائص الرئيسية
- مكتوب بلغة **C** للأداء الأقصى
- يولّد كود C مخصص لكل مسألة (لا يحل المسألة عمومًا بل يبني حلاًّ مخصصًا)
- مصمم للعمل على **أنظمة مدمجة** في الزمن الحقيقي
- يستخدم في مشاريع حقيقية (مثل نظام القيادة الذاتية openpilot)
- مرخص BSD-2 (مفتوح المصدر بالكامل)

## 2. لماذا نحتاج acados؟

### المشكلة في النظام الحالي

وحدة `RocketGNC.cpp` الحالية تستخدم:

**1. جدولة المكاسب (Gain Scheduling):**
```
دالة _pilotGain:
  إذا كان الضغط الديناميكي < 20 kgf/m²:
    استخدم مكاسب المرحلة 1 (Kp=2.0)
  وإلا:
    احسب المكاسب حسب الضغط الديناميكي (Kp = 0.5 / q_dyn)
```
هذه قيم ثابتة مضبوطة يدويًا. إذا تغيرت الظروف بشكل غير متوقع، لا يتكيف النظام.

**2. تحكم PID للارتفاع (المرحلة 1):**
```
خطأ = الارتفاع المطلوب - الارتفاع الحالي
أمر التحكم = Kp × خطأ + Ki × تكامل(خطأ) + Kd × مشتقة(خطأ)
```
PID لا يعرف شيئًا عن مستقبل المسار ولا عن قيود الزعانف.

**3. ملاحة تناسبية (PN) للتوجيه (المرحلة 2):**
```
دالة _pn2 (Biased Proportional Navigation):
  تسارع_مطلوب = N × سرعة_إغلاق × معدل_دوران_خط_الرؤية + تحيز
```
PN جيد لكنه لا يراعي قيود الحمل الأقصى ولا حدود حركة الزعانف.

### ما الذي ينقص؟

| القيد | PID/PN الحالي | acados MPC |
|-------|--------------|------------|
| حدود زاوية الهجوم (مثلاً ±15°) | لا يعرفها، قد يتجاوزها | يضمنها رياضيًا |
| حدود حركة الزعانف (±25°) | لا يعرفها | يحترمها تلقائيًا |
| سرعة حركة الزعانف (°/ثانية) | لا يراعيها | يأخذها بالحسبان |
| عامل الحمل الأقصى (g) | لا يحد منه | يضمن عدم تجاوزه |
| تحسين استهلاك التحكم | لا يوجد | يقلل حركة الزعانف غير الضرورية |
| التنبؤ بالمستقبل | لا يوجد (يتفاعل فقط) | يتنبأ ويخطط للأمام |

### شرح المكونات الرئيسية

#### 3.1 النواة (`acados/`)
هذا هو قلب المكتبة. يحتوي على خوارزميات الحل الرياضي:

- **SQP (Sequential Quadratic Programming):** يحل المسألة غير الخطية بتقسيمها إلى سلسلة مسائل تربيعية. دقيق لكنه أبطأ.
- **SQP-RTI (Real-Time Iteration):** نسخة مبسطة من SQP تنفذ تكرارًا واحدًا فقط في كل دورة تحكم. أسرع بكثير وكافٍ للتحكم اللحظي.
- **DDP (Differential Dynamic Programming):** خوارزمية بديلة للمسائل ذات البنية الخاصة.

#### 3.2 واجهة Python (`interfaces/acados_template/`)
هنا يصف المهندس المسألة بلغة Python:

```
AcadosModel → يعرّف معادلات حركة الصاروخ
     ↓
AcadosOcp → يعرّف دالة التكلفة والقيود
     ↓
AcadosOcpSolver → يولّد كود C مخصص
     ↓
كود C جاهز للدمج مع PX4
```

#### 3.3 BLASFEO و HPIPM (`external/`)
- **BLASFEO:** مكتبة جبر خطي مُحسَّنة للمعالجات المدمجة. أسرع من BLAS/LAPACK القياسية لأنها تستغل بنية المسائل الصغيرة.
- **HPIPM:** حل البرمجة التربيعية باستخدام طريقة النقطة الداخلية. هذا ما يحل المسألة الفرعية في كل تكرار SQP.

#### 3.4 قوالب الكود (`c_templates_tera/`)
عندما تصف المسألة بـ Python، يستخدم acados هذه القوالب لتوليد:
- `acados_solver_M130.c` ← الحل المخصص لصاروخك
- `main_M130.c` ← برنامج اختبار مستقل
- `Makefile` / `CMakeLists.txt` ← ملفات بناء جاهزة

الكود المولَّد **لا يعتمد على Python** ولا على أي إطار عمل. هو كود C صرف يمكن دمجه في أي مكان.

### 3.5 كيف تستفيد ملفاتك من الكود المصدري لـ acados؟

هذا سؤال جوهري: عندما تكتب ملفات Python (النموذج + إعدادات OCP)، هل هي مستقلة أم تستخدم كود acados المصدري؟

**الإجابة: تستخدمه بالكامل.** وإليك بالضبط كيف:

#### ما الذي يحدث عند تشغيل `AcadosOcpSolver(ocp)`؟

هذا السطر الواحد يُفجّر سلسلة عمليات داخل acados:

```
أنت تكتب (Python)              acados يفعل تلقائيًا                 من أين يأخذ الكود
──────────────────              ──────────────────                   ────────────────────

m130_acados_model.py            يأخذ معادلات حركتك الـ 6             CasADi يحسب المشتقات
(معادلات حركة الصاروخ)    →    ويحسب تلقائيًا:                →    (Jacobians) رمزيًا
                                - مصفوفة df/dx (مشتقات بالحالات)     ويكتبها كدوال C في
                                - مصفوفة df/du (مشتقات بالتحكم)      m130_model/m130_expl_ode_fun.c
                                - مكامل RK4 للديناميكا               m130_model/m130_expl_vde_forw.c

m130_ocp_setup.py               يقرأ إعداداتك ويختار                من الكود المصدري:
(تكلفة + قيود + حل)       →    الخوارزميات المناسبة من         →    acados/ocp_nlp/ocp_nlp_sqp_rti.c
                                كوده المصدري:                        acados/dense_qp/dense_qp_hpipm.c
                                - SQP_RTI (لأنك اخترته)              acados/sim/sim_erk_integrator.c
                                - HPIPM (لأنك اخترته)                external/blasfeo/
                                - ERK (لأنك اخترته)                  external/hpipm/

AcadosOcpSolver(ocp)            يستخدم القوالب في                   من القوالب:
(سطر توليد الكود)         →    c_templates_tera/ ويملؤها        →    acados_solver.in.c → acados_solver_m130.c
                                ببيانات مسألتك بالذات                 main.in.c → main_m130.c
                                (أبعاد، أوزان، قيود، ...)             CMakeLists.in.txt → CMakeLists.txt

```

```
    │
    ├── 1. acados_solver_m130.c يستدعي:
    │       ocp_nlp_sqp_rti()           ← من libacados.a
    │       (خوارزمية Real-Time Iteration)
    │
    ├── 2. داخل SQP_RTI:
    │       ├── يحسب الديناميكا:
    │       │     m130_expl_ode_fun()    ← من الكود المولَّد (معادلاتك)
    │       │     sim_erk_integrator()   ← من libacados.a (المكامل RK4)
    │       │
    │       ├── يحسب التكلفة والقيود:
    │       │     m130_cost_y_fun()      ← من الكود المولَّد (أوزانك)
    │       │     m130_constr_h_fun()    ← من الكود المولَّد (قيودك)
    │       │
    │       └── يحل المسألة التربيعية:
    │             hpipm_solve()          ← من libhpipm.a
    │             blasfeo_dgemm()        ← من libblasfeo.a (ضرب مصفوفات سريع)
    │
    └── 3. النتيجة: u_optimal (أفضل أمر تحكم)
```

#### الخلاصة

| ما تكتبه أنت | ما يأتي من acados المصدري |
|---|---|
| معادلات الحركة (7 معادلات: 6 أصلية + تتبع التحكم السابق) | المشتقات التلقائية (Jacobians) + المكامل (RK4) |
| أوزان دالة التكلفة (8 أرقام) | خوارزمية Gauss-Newton + بنية Least Squares |
| القيود (حدود الزعنفة وزاوية الهجوم) | إدارة القيود في NLP + Interior Point Method |
| اختيار الحل: SQP_RTI | خوارزمية SQP-RTI كاملة (~3000 سطر C) |
| اختيار QP: HPIPM | حل QP كامل (~5000 سطر C) + BLASFEO (~20000 سطر) |

**أنت تكتب ~100 سطر Python. acados يربطها بـ ~30,000 سطر C مُحسَّن ومُختبَر.**

لا تعيد اختراع العجلة. تصف ماذا تريد، وacados يبني الحل من كوده المصدري.

### 3.6 توضيح مهم: Python لا يكرر المحاكاة

قد يبدو أن ملفات Python تعيد بناء محاكاة كاملة. **هذا غير صحيح.** لكل ملف دور مختلف تمامًا:

#### أدوار الملفات الثلاثة

```
┌─────────────────────────────────────────────────────────────────────────┐
│  الدور 1: توليد الكود (مرة واحدة فقط على حاسوبك)                      │
│  ══════════════════════════════════════════════                         │
│  m130_acados_model.py ─┐                                               │
│  m130_ocp_setup.py    ─┤→ AcadosOcpSolver(ocp) → c_generated_code/*.c │
│                        │  (تشغّله مرة واحدة)     (هذا فقط ينتقل)      │
│                        │                                               │
│                        └── Python انتهى دوره هنا. لا يُنقل للهاتف.     │
├─────────────────────────────────────────────────────────────────────────┤
│  الدور 2: الاختبار والمقارنة (اختياري، على الحاسوب فقط)                │
│  ════════════════════════════════════════════════════                   │
│  test_mpc_standalone.py  ─┐                                            │
│  run_mpc_comparison.py   ─┤→ للتحقق فقط: "هل MPC أفضل من PID؟"       │
│  m130_mpc_autopilot.py   ─┘  لا تُنقل للهاتف أبدًا.                   │
└─────────────────────────────────────────────────────────────────────────┘
```

#### على الهاتف: لا توجد محاكاة إطلاقًا

```
محاكاة 6DOF (حاسوب فقط):             الهاتف (الواقع):
─────────────────────────             ──────────────────

تحسب فيزياء الصاروخ كاملة:           لا يحسب فيزياء!
 - ايروديناميكا                       المستشعرات تعطيه الحالة الحقيقية
 - جاذبية                             مباشرة (GPS + IMU + بارومتر)
 - دفع محرك
 - 6 درجات حرية
         ↓                                     ↓
تُنتج: حالة مُحاكاة                  يقرأ: حالة حقيقية من uORB
         ↓                                     ↓
MPC يحل → أمر تحكم                   MPC يحل → أمر تحكم
         ↓                                     ↓
يرجع للمحاكاة                        يذهب للزعانف الحقيقية
```

#### MPC ليس محاكاة - هو حل رياضي فقط

ما يفعله acados على الهاتف كل 10ms:

```
المدخل:  6 أرقام من المستشعرات (الحالة الحقيقية الآن)
العملية: حل مسألة تحسين رياضية (جبر خطي + تحسين عددي)
المخرج:  رقم واحد = أفضل أمر تحكم للزعنفة
```

هذا ليس "محاكاة". هذا مثل الآلة الحاسبة: تدخل أرقامًا، تخرج نتيجة.

#### النموذج المبسّط داخل MPC ≠ محاكاة

نعم، MPC يحتوي "نموذجًا" للصاروخ، لكنه يختلف جوهريًا عن المحاكاة:

| | محاكاة 6DOF الكاملة | نموذج MPC الداخلي |
|---|---|---|
| **الغرض** | تقليد الواقع بأعلى دقة | تنبؤ سريع وتقريبي |
| **التعقيد** | 6 درجات حرية + 200 معامل | نموذج مبسّط + 10 معاملات |
| **الزمن** | ثوانٍ لمحاكاة رحلة كاملة | أقل من 5ms لدورة واحدة |
| **يعمل على** | الحاسوب فقط (Python) | الهاتف (كود C مولَّد) |
| **الهدف** | اختبار وتصميم | اتخاذ قرار لحظي |

النموذج داخل MPC هو "خريطة تقريبية" يستخدمها للتنبؤ بالثانيتين القادمتين فقط، ثم يُلغيها ويعيد الحساب مع قراءة جديدة من المستشعرات. لا يحتاج دقة المحاكاة لأنه يُصحّح نفسه كل 10ms.

### 3.7 ماذا يوجد فعلاً على الهاتف: الكود المولَّد + الكود المصدري معًا

### 3.8 هل MPC بديل للتوجيه أم للتحكم أم لكليهما؟

لفهم هذا، يجب أولاً فهم ما يفعله كل جزء في النظام الحالي:

```
النظام الحالي (PID + PN):
══════════════════════════

الهدف → [ التوجيه (PN) ]  →  تسارع مطلوب  →  [ التحكم (PID) ]  →  أمر زعنفة
          Guidance                 ↓                 Control              ↓
        "أحتاج 5g لفوق"      accel_cmd           "أعطي δ=12°"        δ_fin

التوجيه = يحدد أين نريد الذهاب (المسار)
التحكم  = يحدد كيف نحرك الزعنفة (التنفيذ)
```

#### الخيارات الثلاثة الممكنة

```
الخيار 1: MPC يستبدل التحكم فقط
────────────────────────────────
الهدف → [PN] → تسارع → [MPC] → δ_fin
                          ↑
              يحترم قيود الزعنفة
              لكن PN قد يطلب تسارعًا مستحيلاً!

الخيار 2: MPC يستبدل التوجيه فقط
─────────────────────────────────
الهدف → [MPC] → تسارع → [PID] → δ_fin
           ↑
     يخطط مسارًا أمثل
     لكن PID قد لا ينفذه بدقة!

الخيار 3: MPC يدمج التوجيه والتحكم معًا  ★
─────────────────────────────────────────────
الهدف → [MPC] → δ_fin   (مباشرة، بدون وسيط)
           ↑
     يرى الهدف + يعرف حدود الزعنفة + يحسب أفضل أمر
     لا يوجد فصل بين التوجيه والتحكم
```

#### لماذا الخيار 3 هو الأنسب لـ M130؟

```
مشكلة الفصل (الخيار 1 أو 2):
─────────────────────────────
PN يقول: "أحتاج 30g لفوق!"
التحكم يرد: "مستحيل! الحد 20g والزعنفة وصلت 25°!"
النتيجة: تشبّع (saturation) → فقدان الدقة → زيادة Miss Distance

ميزة الدمج (الخيار 3):
──────────────────────
MPC يرى: "الهدف هناك. حدودي: 20g، ±25° زعنفة، ±15° زاوية هجوم"
MPC يحسب: "أبدأ المناورة الآن بـ 12g بدلاً من الانتظار ثم طلب 30g"
النتيجة: مسار أمثل لا يتجاوز أي حد ✓
```

| المقارنة | فصل التوجيه عن التحكم | دمجهما في MPC |
|---|---|---|
| PN يطلب المستحيل؟ | نعم، ثم التحكم يشبّع | لا، MPC يعرف حدوده من البداية |
| التخطيط المسبق | PN يتفاعل فقط (لا تنبؤ) | MPC يتنبأ لثانيتين قادمتين |
| التكامل بين المحاور | pitch و yaw مستقلان | يحلهما معًا كمسألة واحدة |
| ضمان القيود | قد تُنتهك بين الطبقتين | مضمونة رياضيًا |

#### كيف يظهر هذا في الكود؟

الكود في الدليل مصمم بالفعل وفق **الخيار 3** (دمج التوجيه والتحكم):

```python
# متجه الحالة يتضمن متغيرات التوجيه والتحكم معًا:
x = [V, gamma, q_rate, alpha, h, x_pos]
#         ↑        ↑       ↑
#      توجيه    تحكم    تحكم
#   (اتجاه المسار) (سلوك الصاروخ)

# المخرج: أمر الزعنفة مباشرة (ليس تسارعًا مطلوبًا)
u = [delta_e]   ← أمر الزعنفة، ليس أمر توجيه

# دالة التكلفة تجمع أهداف التوجيه والتحكم:
W_gamma = 100.0   ← توجيه: "اتجه نحو الهدف"
W_h     = 50.0    ← توجيه: "حافظ على الارتفاع المطلوب"
W_q     = 1.0     ← تحكم: "قلل معدل الدوران"
W_alpha = 10.0    ← تحكم: "قلل زاوية الهجوم"
W_delta = 0.1     ← تحكم: "قلل حركة الزعنفة"
```

MPC يأخذ **موقع الهدف** كمدخل ويُخرج **أمر الزعنفة** مباشرة. لا يوجد PN في الوسط.

#### لكن PN + PID يبقيان كاحتياط (أمان)

```
في RocketGNC.cpp (المرحلة 2):

float mpc_result = _mpcSolve(حالة_الصاروخ, الهدف);

if (نجح mpc_result) {
    // MPC يتولى التوجيه + التحكم معًا
    delta_fin = mpc_result;
} else {
    // احتياط: النظام التقليدي
    accel_cmd = _pn2(الهدف);         // PN للتوجيه
    delta_fin = _autopilot(accel_cmd); // PID للتحكم
}
```

**الخلاصة: MPC يدمج التوجيه والتحكم في حل واحد. هذه هي قوته الحقيقية. PN + PID يبقيان فقط كشبكة أمان.**

## 4. كيف يعمل acados خطوة بخطوة

### المرحلة الأولى: التصميم (على الحاسوب، مرة واحدة)

```
الخطوة 1: تعريف النموذج الديناميكي بـ CasADi
──────────────────────────────────────────────
import casadi as ca
from acados_template import AcadosModel

# حالات الصاروخ (ما نريد تتبعه)
x = ca.SX.sym('x', 6)  # [موقع_x, موقع_y, موقع_z, سرعة, زاوية_الميل, معدل_الميل]

# أوامر التحكم (ما نتحكم به)
u = ca.SX.sym('u', 2)  # [انحراف_الزعنفة_ميل, انحراف_الزعنفة_انعراج]

# معادلات الحركة (كيف يتحرك الصاروخ)
f_expl = ديناميكا_الصاروخ(x, u)  # من 6DOF model

model = AcadosModel()
model.x = x
model.u = u
model.f_expl_expr = f_expl
```

```
الخطوة 2: تعريف مسألة التحكم الأمثل
──────────────────────────────────────
from acados_template import AcadosOcp

ocp = AcadosOcp()
ocp.model = model

# ماذا نريد تحقيقه؟ (دالة التكلفة)
ocp.cost  → تقليل الخطأ عن المسار المطلوب + تقليل حركة الزعانف

# ما القيود التي يجب احترامها؟
ocp.constraints → زاوية الهجوم ≤ 15°
                → انحراف الزعنفة ≤ 25°
                → عامل الحمل ≤ 20g
                → سرعة الزعنفة ≤ 200°/ثانية

# إعدادات الحل
ocp.solver_options.tf = 2.0            # أفق التنبؤ = 2 ثانية
ocp.solver_options.N_horizon = 20      # 20 خطوة في الأفق
ocp.solver_options.nlp_solver_type = 'SQP_RTI'  # للزمن الحقيقي
ocp.solver_options.qp_solver = 'PARTIAL_CONDENSING_HPIPM'
```

```
الخطوة 3: توليد كود C
──────────────────────
from acados_template import AcadosOcpSolver

AcadosOcpSolver.generate(ocp, json_file='m130_ocp.json')
AcadosOcpSolver.build(ocp.code_export_directory)

# النتيجة: مجلد c_generated_code/ يحتوي على كود C جاهز
```

```
في كل دورة تحكم (كل 10ms):
┌─────────────────────────────────────────────────┐
│ 1. اقرأ حالة الصاروخ الحالية من uORB            │
│    (موقع، سرعة، اتجاه، معدلات دوران)           │
│                                                  │
│ 2. حدّث حالة الحل:                              │
│    ocp_solver_set(solver, 0, "x", x_current)     │
│                                                  │
│ 3. حل المسألة:                                   │
│    status = ocp_solver_solve(solver)              │
│    ← هنا يحسب acados أفضل سلسلة أوامر تحكم     │
│    ← لـ 20 خطوة قادمة (2 ثانية في المستقبل)     │
│                                                  │
│ 4. خذ أول أمر فقط:                               │
│    ocp_solver_get(solver, 0, "u", u_optimal)      │
│                                                  │
│ 5. أرسل الأمر للزعانف عبر uORB                  │
│                                                  │
│ 6. في الدورة التالية: كرر من الخطوة 1            │
│    (بحالة جديدة محدثة = هذا هو MPC)              │
└─────────────────────────────────────────────────┘
```

### لماذا هذا أفضل من PID؟

```
PID (الحالي):
الدورة 1: أرى خطأ 5° → أعطي أمر تصحيح → أتمنى أن يعمل
الدورة 2: أرى خطأ 3° → أعطي أمر تصحيح → أتمنى أن يعمل
(لا يعرف ماذا سيحدث بعد لحظة)

MPC (acados):
الدورة 1: أرى حالة الصاروخ الآن
         → أحسب أفضل 20 أمر للثانيتين القادمتين
         → أختار فقط أول أمر (الأمثل الآن)
         → أضمن أن كل الأوامر تحترم القيود
الدورة 2: أرى الحالة الجديدة (التي تغيرت بسبب الأمر السابق + اضطرابات)
         → أعيد الحساب بالكامل مع المعلومات الجديدة
(يتكيف مع أي تغيير لأنه يعيد الحساب كل مرة)
```

## 5. ماذا سيقدم لصاروخ M130

### 5.1 التحكم في المسار الأمثل (Optimal Trajectory Tracking)

**الوضع الحالي:**
- المرحلة 1: PID يحاول تثبيت الارتفاع → قد يتأرجح (overshoot)
- المرحلة 2: PN يوجه نحو الهدف → لا يعرف حدود الصاروخ

**مع acados:**
- يحسب المسار الأمثل من الوضع الحالي إلى الهدف
- يراعي ديناميكا الصاروخ الفعلية (ليس تقريبًا خطيًا)
- يتجنب المناورات العنيفة غير الضرورية

### 5.2 احترام القيود الفيزيائية

acados يضمن رياضيًا أن الأوامر لن تتجاوز الحدود:

```
القيود التي يمكن تعريفها:
├── قيود على أوامر التحكم (u):
│   ├── -25° ≤ زاوية الزعنفة ≤ +25°
│   └── -200°/s ≤ سرعة الزعنفة ≤ +200°/s
│
├── قيود على حالة الصاروخ (x):
│   ├── زاوية الهجوم |α| ≤ 15°
│   ├── عامل الحمل ≤ 20g
│   └── زاوية الانزلاق |β| ≤ 10°
│
└── قيود على المسار:
    ├── الارتفاع الأدنى ≥ حد أمان
    └── مسافة التقاطع مع الهدف ≤ قيمة محددة
```

### 5.3 تعويض انحراف محور الدفع (Thrust Misalignment)

مشروعك يدرس هذه المشكلة فعلاً في `run_misalignment_test.py`.

**الوضع الحالي:** PID يتفاعل مع الانحراف بعد حدوثه.
**مع acados:** يُدخل الانحراف المقاس كمعامل في النموذج، فيعوّض عنه بشكل استباقي ومثالي عبر المحاور الثلاثة معًا.

### 5.4 تحكم متعدد المتغيرات (MIMO)

**الوضع الحالي في RocketGNC:**
```
ثلاث حلقات تحكم مستقلة:
  حلقة الميل (Pitch)  → تحسب أمر الزعنفة للميل
  حلقة الانعراج (Yaw)  → تحسب أمر الزعنفة للانعراج
  حلقة اللف (Roll)    → تحسب أمر الزعنفة للف

كل حلقة لا تعرف ماذا تفعل الأخرى
← قد تتعارض الأوامر
```

**مع acados:**
```
حل واحد متكامل:
  يحسب أوامر الميل والانعراج واللف معًا
  ← يضمن عدم التعارض
  ← يوزع جهد التحكم بالشكل الأمثل
  ← يعوض التقارن بين المحاور (cross-coupling)
```

### 5.5 تحسين المرحلة النهائية (Terminal Guidance)

في آخر ثوانٍ قبل الوصول للهدف، الدقة حاسمة:

**PN الحالي:** يتبع خط الرؤية بمكسب ثابت N.
**acados MPC:** يحسب مسار الاعتراض الأمثل مع مراعاة:
- سرعة الإغلاق المتغيرة
- القدرة المتبقية للمناورة
- الحد الأدنى لمسافة الخطأ عند التقاطع (Miss Distance)

### 5.6 جدول ملخص الفوائد

| الجانب | PID + PN (الحالي) | acados MPC (المطوّر) | التحسن المتوقع |
|--------|------------------|---------------------|----------------|
| دقة التوجيه | جيدة | ممتازة | تقليل Miss Distance بـ 30-60% |
| احترام القيود | غير مضمون | مضمون رياضيًا | صفر تجاوزات |
| تعويض الاضطرابات | تفاعلي (بعد الحدوث) | استباقي (يتنبأ) | استجابة أسرع |
| استهلاك التحكم | غير محسَّن | أمثل | عمر أطول للزعانف |
| التكيف مع الظروف | يحتاج ضبط يدوي | تلقائي | مرونة أعلى |
| الأمان | لا ضمان | قيود مضمونة | موثوقية أعلى |

## 8. أمثلة تطبيقية على M130

### مثال 1: تحسين المرحلة 1 (تثبيت الارتفاع)

```
المشكلة الحالية:
  PID يعطي overshoot عند الوصول للارتفاع المطلوب
  (يرتفع أكثر من اللازم ثم يرجع)

حل MPC:
  يعرف أن الصاروخ ثقيل ويحتاج وقتًا للتباطؤ
  → يبدأ بتقليل الدفع قبل الوصول للارتفاع
  → وصول سلس بدون تجاوز
```

### مثال 2: التوجيه مع قيود g

```
المشكلة الحالية:
  PN قد يطلب مناورة 30g لتصحيح مسار حاد
  → تجاوز الحد الهيكلي → تلف الصاروخ

حل MPC:
  يعرف أن الحد 20g
  → يبدأ المناورة مبكرًا بتسارع أقل
  → يصل لنفس النتيجة بدون تجاوز الحد
```

### مثال 3: تعويض انحراف الدفع

```
المشكلة الحالية:
  انحراف محور الدفع 1° يسبب لحظة ميل ثابتة
  PID يعوض لكن يستهلك مدى حركة الزعنفة
  → يبقى مدى أقل للمناورة الفعلية

حل MPC:
  يُدخل الانحراف كمعامل ثابت في النموذج
  → يحسب الانحراف المطلوب للتعويض + التوجيه معًا
  → يوزع الجهد بالشكل الأمثل
```

### مثال 4: التوجيه النهائي (آخر 3 ثوانٍ)

```
المشكلة الحالية:
  PN بمكسب N=4 ثابت
  في اللحظات الأخيرة، الخطأ الصغير يتضخم
  → Miss Distance قد تكون كبيرة

حل MPC:
  أفق التنبؤ يتقلص تلقائيًا مع اقتراب الهدف
  → يكثف الحساب في اللحظات الحرجة
  → يأخذ بالحسبان القدرة المتبقية للمناورة
  → يقلل Miss Distance بشكل كبير
```

### 9.2 الخطوة 1: كتابة نموذج الصاروخ بـ CasADi

أنشئ ملفًا جديدًا `6DOF_v2_130/mpc/m130_acados_model.py`:

```python
import casadi as ca
import numpy as np
from acados_template import AcadosModel

def create_m130_model():
    model = AcadosModel()
    model.name = "m130_rocket"

    # ──────────────────────────────────────────────
    # تعريف متغيرات الحالة (ما نتتبعه)
    # ──────────────────────────────────────────────
    # نستخدم نموذجًا مبسطًا لمستوى الميل (Pitch Plane) كبداية
    # هذا أسهل للفهم والاختبار قبل التوسع لثلاثة محاور

    V     = ca.SX.sym('V')         # السرعة الكلية (m/s)
    gamma = ca.SX.sym('gamma')     # زاوية مسار الطيران (rad)
    q_rate= ca.SX.sym('q_rate')    # معدل الميل (rad/s)
    alpha = ca.SX.sym('alpha')     # زاوية الهجوم (rad)
    h     = ca.SX.sym('h')         # الارتفاع (m)
    x_pos = ca.SX.sym('x_pos')     # المسافة الأفقية (m)

    x = ca.vertcat(V, gamma, q_rate, alpha, h, x_pos)
    # عدد الحالات = 6

    # ──────────────────────────────────────────────
    # تعريف أوامر التحكم (ما نتحكم به)
    # ──────────────────────────────────────────────
    delta_e = ca.SX.sym('delta_e')  # انحراف زعنفة الميل (rad)

    u = ca.vertcat(delta_e)
    # عدد أوامر التحكم = 1

    # ──────────────────────────────────────────────
    # تعريف المعاملات (قيم تتغير كل دورة لكن ليست حالات)
    # ──────────────────────────────────────────────
    m      = ca.SX.sym('m')        # الكتلة الحالية (kg)
    thrust = ca.SX.sym('thrust')   # قوة الدفع (N)
    rho    = ca.SX.sym('rho')      # كثافة الهواء (kg/m³)
    Iyy    = ca.SX.sym('Iyy')      # عزم القصور الذاتي (kg.m²)

    p = ca.vertcat(m, thrust, rho, Iyy)

    # ──────────────────────────────────────────────
    # الثوابت الهندسية للصاروخ (من بيانات CFD)
    # ──────────────────────────────────────────────
    S_ref = 0.0133        # المساحة المرجعية (m²) - مقطع الصاروخ
    d_ref = 0.130         # القطر المرجعي (m)
    g = 9.81

    # ──────────────────────────────────────────────
    # المعاملات الايروديناميكية (تقريب خطي كبداية)
    # ──────────────────────────────────────────────
    # هذه قيم تقريبية مأخوذة من بيانات CFD في rocket_M130_v01_CFD68
    # لاحقًا يمكن استبدالها بجداول أدق أو تقريب متعدد الحدود

    CL_alpha = 8.0        # ميل منحنى الرفع (1/rad)
    CL_delta = 4.0        # فعالية الزعنفة في الرفع (1/rad)
    CD_0     = 0.4         # مقاومة الاحتكاك الأساسية
    CD_alpha2= 2.0         # مقاومة مستحثة بزاوية الهجوم
    Cm_alpha = -3.0        # ميل لحظة الميل (1/rad) - سالب = مستقر
    Cm_delta = -5.0        # فعالية الزعنفة في لحظة الميل (1/rad)
    Cm_q     = -10.0       # تخميد معدل الميل (1/rad)

    # ──────────────────────────────────────────────
    # حساب القوى والعزوم
    # ──────────────────────────────────────────────
    q_dyn = 0.5 * rho * V**2              # الضغط الديناميكي (Pa)

    # القوى الايروديناميكية
    CL = CL_alpha * alpha + CL_delta * delta_e
    CD = CD_0 + CD_alpha2 * alpha**2
    L = q_dyn * S_ref * CL                # قوة الرفع (N)
    D = q_dyn * S_ref * CD                # قوة السحب (N)

    # لحظة الميل
    Cm = Cm_alpha * alpha + Cm_delta * delta_e + Cm_q * (q_rate * d_ref / (2 * V))
    M_pitch = q_dyn * S_ref * d_ref * Cm  # لحظة الميل (N.m)

    # ──────────────────────────────────────────────
    # معادلات الحركة (6 معادلات تفاضلية)
    # ──────────────────────────────────────────────
    V_dot     = (thrust * ca.cos(alpha) - D - m * g * ca.sin(gamma)) / m
    gamma_dot = (thrust * ca.sin(alpha) + L - m * g * ca.cos(gamma)) / (m * V)
    q_dot     = M_pitch / Iyy
    alpha_dot = q_rate - gamma_dot
    h_dot     = V * ca.sin(gamma)
    x_dot     = V * ca.cos(gamma)

    f_expl = ca.vertcat(V_dot, gamma_dot, q_dot, alpha_dot, h_dot, x_dot)

    # ──────────────────────────────────────────────
    # تجميع النموذج
    # ──────────────────────────────────────────────
    model.x = x
    model.u = u
    model.p = p
    model.f_expl_expr = f_expl

    return model
```

**ملاحظة مهمة:** هذا نموذج مبسط لمستوى الميل فقط (2D). الهدف هو اختباره أولاً ثم التوسع لثلاثة محاور.

### 9.3 الخطوة 2: تعريف مسألة التحكم الأمثل (متعدد المراحل)

الصاروخ M130 يمر بثلاث مراحل مختلفة جذريًا، كل مرحلة لها فيزياء وأولويات مختلفة:

```
المرحلة       الخطوات   الدفع    الأولوية الرئيسية
──────        ───────   ─────    ────────────────
1: إطلاق     5 خطوات   3000N    ثبات + منع الدوران (المحرك يعمل بقوة)
2: صعود      10 خطوات  0N       تتبع مسار الملاحة (طيران حر)
3: نهائي     5 خطوات   0N       دقة الإصابة (اللحظة الحاسمة)
```

لذلك نستخدم **Multi-Phase OCP** كنهج رسمي: نموذج واحد بأوزان وقيود مخصصة لكل مرحلة.

أنشئ ملف `6DOF_v2_130/mpc/m130_ocp_setup.py`:

```python
import numpy as np
import casadi as ca
from acados_template import AcadosMultiphaseOcp, AcadosOcp, AcadosOcpSolver
from m130_acados_model import create_m130_model

def create_m130_ocp():
    """
    MPC متعدد المراحل لصاروخ M130
    3 مراحل: إطلاق (5 خطوات) → صعود (10 خطوات) → توجيه نهائي (5 خطوات)
    كل مرحلة لها أوزان وقيود مخصصة حسب أولوياتها

    ملاحظة: النموذج يُوسَّع بحالة إضافية delta_e_prev لفرض قيود
    على معدل تغيّر التحكم (control rate). acados لا يدعم lbu_rate
    مباشرة، لذا نضيف delta_e_prev كحالة ونقيّد الفرق
    (delta_e - delta_e_prev) عبر قيد غير خطي.
    """
    model = create_m130_model()

    # ──────────────────────────────────────────────
    # توسيع النموذج: إضافة حالة delta_e_prev لقيود معدل التحكم
    # ──────────────────────────────────────────────
    # acados لا يوفر lbu_rate / ubu_rate مباشرة.
    # الطريقة الصحيحة: إضافة أمر التحكم السابق كحالة إضافية
    # ثم تقييد الفرق (u - x_prev) كقيد غير خطي (nonlinear constraint).
    delta_e_prev = ca.SX.sym('delta_e_prev')

    # الحالة الموسّعة: 6 أصلية + 1 للتحكم السابق = 7
    x_aug = ca.vertcat(model.x, delta_e_prev)

    # معادلة تطور delta_e_prev: في الخطوة التالية تصبح u الحالي
    f_expl_aug = ca.vertcat(model.f_expl_expr, model.u[0])

    model.x = x_aug
    model.f_expl_expr = f_expl_aug

    # ──────────────────────────────────────────────
    # الأبعاد المشتركة (بعد التوسيع)
    # ──────────────────────────────────────────────
    nx = 7                # عدد الحالات: V, gamma, q, alpha, h, x, delta_e_prev
    nu = 1                # عدد أوامر التحكم: delta_e
    np_param = 4          # عدد المعاملات: mass, thrust, rho, Iyy
    N_total = 20          # إجمالي خطوات الأفق

    x = model.x
    u = model.u

    # ══════════════════════════════════════════════
    # دالة مساعدة: إعداد مرحلة واحدة
    # ══════════════════════════════════════════════
    def setup_phase(N_phase, W_diag, W_e_diag, delta_max_deg, alpha_max_deg,
                    delta_rate_max_deg_per_s=100.0):
        """إعداد OCP لمرحلة واحدة بأوزان وقيود مخصصة"""
        phase = AcadosOcp()
        phase.model = model
        phase.dims.N = N_phase

        # ── التكلفة ──
        phase.cost.cost_type = 'NONLINEAR_LS'
        phase.cost.cost_type_e = 'NONLINEAR_LS'
        # 7 حالات (6 أصلية + delta_e_prev) + 1 تحكم = 8
        phase.model.cost_y_expr = ca.vertcat(x, u)
        phase.model.cost_y_expr_e = x
        phase.cost.yref = np.zeros(nx + nu)
        phase.cost.yref_e = np.zeros(nx)
        phase.cost.W = np.diag(W_diag)
        phase.cost.W_e = np.diag(W_e_diag)

        # ── قيود الزعانف ──
        delta_max = np.radians(delta_max_deg)
        phase.constraints.lbu = np.array([-delta_max])
        phase.constraints.ubu = np.array([+delta_max])
        phase.constraints.idxbu = np.array([0])

        # ── Control Rate: منع أوامر مفاجئة ──
        # acados لا يدعم lbu_rate / ubu_rate مباشرة.
        # بدلاً من ذلك نستخدم قيد غير خطي على الفرق:
        #   h(x, u) = u[0] - x[6]  (أي delta_e - delta_e_prev)
        # ونقيّد: -delta_rate_max*dt ≤ h ≤ +delta_rate_max*dt
        delta_rate_max = np.radians(delta_rate_max_deg_per_s)
        dt_avg = 0.01  # متوسط الخطوة الزمنية

        h_rate = u[0] - x[6]  # x[6] = delta_e_prev (الحالة المُضافة)
        phase.model.con_h_expr = h_rate
        phase.constraints.lh = np.array([-delta_rate_max * dt_avg])
        phase.constraints.uh = np.array([+delta_rate_max * dt_avg])

        # ── قيود زاوية الهجوم ──
        alpha_max = np.radians(alpha_max_deg)
        phase.constraints.lbx = np.array([-alpha_max])
        phase.constraints.ubx = np.array([+alpha_max])
        phase.constraints.idxbx = np.array([3])  # alpha هي الحالة رقم 3

        # ── Soft Constraints: قيود مرنة تمنع فشل الحل ──
        # نجعل قيد alpha مرناً (soft) + قيد control rate مرناً أيضاً
        phase.constraints.idxsbx = np.array([0])     # alpha soft
        phase.constraints.idxsh = np.array([0])      # control rate soft
        phase.cost.Zl = np.array([1000.0, 500.0])    # عقوبة تربيعية (alpha, rate)
        phase.cost.Zu = np.array([1000.0, 500.0])
        phase.cost.zl = np.array([100.0, 50.0])      # عقوبة خطية (alpha, rate)
        phase.cost.zu = np.array([100.0, 50.0])

        # ── المعاملات ──
        phase.parameter_values = np.array([12.0, 3000.0, 1.225, 0.05])

        return phase

    # ══════════════════════════════════════════════
    # المرحلة 1: الإطلاق (5 خطوات، دفع نشط 3000N)
    # ══════════════════════════════════════════════
    # الأولوية: منع الدوران + تثبيت الميل
    # المحرك يعمل بقوة → أي خطأ يتضخم بسرعة
    phase1 = setup_phase(
        N_phase=5,
        W_diag=[
            0.1,     # V:            ليس مهمًا (المحرك يتحكم بالسرعة)
            10.0,    # gamma:        مهم (زاوية الصعود الصحيحة)
            100.0,   # q:            ★ الأهم (منع الدوران أثناء الدفع)
            50.0,    # alpha:        مهم (منع زاوية هجوم كبيرة)
            0.01,    # h:            غير مهم (الارتفاع يأتي تلقائيًا)
            0.0,     # x:            لا يهم
            0.0,     # delta_e_prev: حالة مساعدة (لا وزن عليها)
            10.0,    # delta_e:      تقليل استخدام الزعانف
        ],
        W_e_diag=[0.1, 10.0, 100.0, 50.0, 0.01, 0.0, 0.0],
        delta_max_deg=15.0,   # ±15° فقط (تحفظ أكثر مع الدفع)
        alpha_max_deg=10.0,   # ±10° (أكثر تحفظًا)
    )

    # ══════════════════════════════════════════════
    # المرحلة 2: الصعود / الطيران الحر (10 خطوات، بدون دفع)
    # ══════════════════════════════════════════════
    # الأولوية: تتبع مسار الملاحة (gamma المرجعي)
    # بدون دفع → أكثر أمانًا → قيود أوسع
    phase2 = setup_phase(
        N_phase=10,
        W_diag=[
            1.0,     # V:            متوسط
            50.0,    # gamma:        ★ الأهم (تتبع مسار الملاحة)
            10.0,    # q:            متوسط (أقل خطورة بدون دفع)
            20.0,    # alpha:        مهم (كفاءة ديناميكية هوائية)
            5.0,     # h:            مهم (الارتفاع المستهدف)
            0.0,     # x:            لا يهم بعد
            0.0,     # delta_e_prev: حالة مساعدة (لا وزن عليها)
            1.0,     # delta_e:      حرية أكبر
        ],
        W_e_diag=[1.0, 50.0, 10.0, 20.0, 5.0, 0.0, 0.0],
        delta_max_deg=25.0,   # ±25° (حرية كاملة)
        alpha_max_deg=15.0,   # ±15° (عادي)
    )

    # ══════════════════════════════════════════════
    # المرحلة 3: التوجيه النهائي (5 خطوات، دقة الإصابة)
    # ══════════════════════════════════════════════
    # الأولوية: دقة الإصابة هي كل شيء
    # آخر 1-2 ثانية → كل خطأ الآن = خطأ في الهدف
    phase3 = setup_phase(
        N_phase=5,
        W_diag=[
            0.1,     # V:            لا يهم
            100.0,   # gamma:        ★ حرج (زاوية الاقتراب)
            5.0,     # q:            أقل أهمية
            10.0,    # alpha:        متوسط
            200.0,   # h:            ★ الأهم (الارتفاع بدقة)
            100.0,   # x:            ★ الأهم (المسافة بدقة)
            0.0,     # delta_e_prev: حالة مساعدة (لا وزن عليها)
            0.1,     # delta_e:      لا تقييد (استخدم كل شيء)
        ],
        W_e_diag=[0.1, 200.0, 5.0, 10.0, 500.0, 500.0, 0.0],  # نهاية: دقة قصوى
        delta_max_deg=25.0,   # حرية كاملة
        alpha_max_deg=20.0,   # أوسع (نحتاج مناورة حادة)
    )

    # ══════════════════════════════════════════════
    # تجميع المراحل الثلاث في مسألة واحدة
    # ══════════════════════════════════════════════
    ocp = AcadosMultiphaseOcp(N_list=[5, 10, 5])
    ocp.set_phase(phase1, 0)   # المرحلة 1: خطوات 0-4
    ocp.set_phase(phase2, 1)   # المرحلة 2: خطوات 5-14
    ocp.set_phase(phase3, 2)   # المرحلة 3: خطوات 15-19

    # الحالة الابتدائية (تُحدَّث في كل دورة بالقيمة الفعلية)
    # 7 حالات: 6 أصلية + delta_e_prev (آخر أمر تحكم مُنفَّذ)
    ocp.constraints.x0 = np.array([170.0, 0.5, 0.0, 0.0, 100.0, 0.0, 0.0])
    #                               V     gamma  q    alpha  h      x   delta_e_prev

    # ──────────────────────────────────────────────
    # إعدادات الحل المشتركة (تُطبق على كل المراحل)
    # ──────────────────────────────────────────────
    ocp.solver_options.nlp_solver_type = 'SQP_RTI'
    # SQP_RTI = Real-Time Iteration
    # يكفي تكرار واحد في كل دورة تحكم

    # ── AS-RTI: تحضير مبكر = استجابة أسرع ──
    # يقسم العمل: التحضير بين الدورات + Feedback عند القياس
    ocp.solver_options.as_rti_iter = 1        # تفعيل AS-RTI
    ocp.solver_options.as_rti_level = 4       # أقصى تقسيم
    # النتيجة: زمن الاستجابة ~0.5ms بدل ~2ms

    ocp.solver_options.qp_solver = 'PARTIAL_CONDENSING_HPIPM'
    ocp.solver_options.hessian_approx = 'GAUSS_NEWTON'
    ocp.solver_options.integrator_type = 'ERK'
    ocp.solver_options.sim_method_num_stages = 4
    ocp.solver_options.nlp_solver_max_iter = 1

    # ── Solver Timeout ──
    ocp.solver_options.nlp_solver_tol_stat = 1e-3

    # ── Non-Uniform Grid: دقة أعلى في البداية، أفق أطول في النهاية ──
    # نريد أفقاً إجمالياً ~2 ثانية (كما صُمم النظام)
    # خطوات قصيرة في البداية (حرجة، أوامر دقيقة) وطويلة في النهاية (تنبؤ بعيد)
    time_steps = np.zeros(N_total)
    time_steps[:5]   = 0.02    # مرحلة 1: 20ms لكل خطوة (دقة عالية أثناء الدفع)
    time_steps[5:15] = 0.10    # مرحلة 2: 100ms (أفق متوسط للتتبع)
    time_steps[15:]  = 0.20    # مرحلة 3: 200ms (أفق أطول لرؤية الهدف مبكرًا)
    ocp.solver_options.time_steps = time_steps
    # المجموع = 5×0.02 + 10×0.10 + 5×0.20 = 0.1 + 1.0 + 1.0 = 2.1 ثانية أفق

    return ocp

def build_solver():
    """بناء الحل متعدد المراحل وتوليد كود C"""
    ocp = create_m130_ocp()
    solver = AcadosOcpSolver(ocp, json_file='m130_ocp.json')
    print("تم بناء الحل متعدد المراحل بنجاح!")
    print(f"الملفات المولّدة في: c_generated_code/")
    print(f"المراحل: إطلاق(5) + صعود(10) + نهائي(5) = 20 خطوة")
    return solver

if __name__ == '__main__':
    build_solver()
```

```
نموذج واحد (أوزان ثابتة):              Multi-Phase (أوزان لكل مرحلة):
──────────────────────                 ─────────────────────────────
أثناء الإطلاق:                         أثناء الإطلاق:
  q_rate وزن 1.0 (قليل!)                 q_rate وزن 100.0 ★ (منع الدوران)
  h وزن 50.0 (مبالغة!)                   h وزن 0.01 (لا يهم بعد)

أثناء الإصابة:                          أثناء الإصابة:
  q_rate وزن 1.0 (لا يهم)                q_rate وزن 5.0 (عادي)
  h وزن 50.0 (قليل!)                     h وزن 200.0 ★ (الدقة هي كل شيء)

→ تنازلات دائمة                        → أداء أمثل في كل لحظة
```

### 9.4 الخطوة 3: اختبار MPC بشكل مستقل

```python
"""
اختبار مستقل لـ MPC متعدد المراحل - يحاكي مسار بسيط ويرسم النتائج
"""
import numpy as np
import matplotlib.pyplot as plt
from m130_ocp_setup import create_m130_ocp
from acados_template import AcadosOcpSolver

def simulate_mpc():
    # بناء الحل متعدد المراحل
    ocp = create_m130_ocp()
    solver = AcadosOcpSolver(ocp, json_file='m130_test.json')
    N = 20  # إجمالي الخطوات (5+10+5)

    # ──────────────────────────────────────────────
    # إعدادات المحاكاة
    # ──────────────────────────────────────────────
    dt_sim = 0.01         # خطوة المحاكاة = 10ms (مثل RocketGNC)
    T_total = 10.0        # مدة المحاكاة الكلية
    N_sim = int(T_total / dt_sim)

    # الحالة الابتدائية
    # [V, gamma, q_rate, alpha, h, x_pos, delta_e_prev]
    x0 = np.array([170.0, 0.3, 0.0, 0.0, 80.0, 0.0, 0.0])

    # المسار المرجعي (الهدف)
    # 7 حالات + 1 تحكم = 8 عناصر
    # نريد: gamma=0.5 rad (~28 درجة)، alpha=0، h=100m، delta_e_prev=0 (مرجع)
    y_ref = np.array([170.0, 0.5, 0.0, 0.0, 100.0, 0.0, 0.0, 0.0])
    y_ref_e = np.array([170.0, 0.5, 0.0, 0.0, 100.0, 0.0, 0.0])

    # المعاملات (كتلة، دفع، كثافة، عزم قصور)
    # نبدأ بمحرك يعمل ثم ينطفئ بعد 4 ثوانٍ
    params_burn = np.array([12.0, 3000.0, 1.225, 0.05])
    params_coast = np.array([8.0, 0.0, 1.225, 0.05])  # بعد الاحتراق: كتلة أقل، دفع = 0

    # ──────────────────────────────────────────────
    # حلقة المحاكاة
    # ──────────────────────────────────────────────
    nx = 7  # عدد الحالات بعد التوسيع
    x_history = np.zeros((N_sim + 1, nx))
    u_history = np.zeros((N_sim, 1))
    t_history = np.zeros(N_sim)
    solve_time_history = np.zeros(N_sim)

    x_current = x0.copy()
    x_history[0, :] = x_current

    for i in range(N_sim):
        t = i * dt_sim

        # اختيار المعاملات حسب مرحلة الطيران الفعلية
        params = params_burn if t < 4.0 else params_coast

        # ── تحديث المسار المرجعي لكل خطوة (المراحل مُدمجة تلقائيًا) ──
        # الحل متعدد المراحل يعرف أوزان كل خطوة مسبقًا
        # نحتاج فقط تحديث yref و p
        for k in range(N):
            solver.set(k, "yref", y_ref)
            solver.set(k, "p", params)
        solver.set(N, "yref", y_ref_e)
        solver.set(N, "p", params)

        # ── تعيين الحالة الحالية ──
        solver.set(0, "lbx", x_current)
        solver.set(0, "ubx", x_current)

        # ── حل المسألة ──
        status = solver.solve()
        solve_time = solver.get_stats("time_tot")

        if status != 0:
            print(f"t={t:.3f}s: الحل فشل (status={status}), نستخدم u=0")
            u_opt = np.array([0.0])
        else:
            u_opt = solver.get(0, "u")

        # ── تسجيل ──
        u_history[i, :] = u_opt
        t_history[i] = t
        solve_time_history[i] = solve_time * 1000  # تحويل إلى مللي ثانية

        # ── تحديث الحالة (محاكاة بسيطة بـ Euler) ──
        # في الاختبار الحقيقي سنستخدم محاكاة 6DOF الكاملة
        x_next = solver.get(1, "x")  # acados يتنبأ الحالة التالية
        x_current = x_next
        x_history[i + 1, :] = x_current

    # ──────────────────────────────────────────────
    # رسم النتائج
    # ──────────────────────────────────────────────
    fig, axes = plt.subplots(4, 2, figsize=(14, 12))
    fig.suptitle('اختبار MPC لصاروخ M130', fontsize=14)

    # السرعة
    axes[0, 0].plot(t_history, x_history[:-1, 0])
    axes[0, 0].set_ylabel('V (m/s)')
    axes[0, 0].set_title('السرعة')
    axes[0, 0].grid(True)

    # زاوية مسار الطيران
    axes[0, 1].plot(t_history, np.degrees(x_history[:-1, 1]))
    axes[0, 1].axhline(y=np.degrees(0.5), color='r', linestyle='--', label='المرجع')
    axes[0, 1].set_ylabel('gamma (deg)')
    axes[0, 1].set_title('زاوية مسار الطيران')
    axes[0, 1].legend()
    axes[0, 1].grid(True)

    # معدل الميل
    axes[1, 0].plot(t_history, np.degrees(x_history[:-1, 2]))
    axes[1, 0].set_ylabel('q (deg/s)')
    axes[1, 0].set_title('معدل الميل')
    axes[1, 0].grid(True)

    # زاوية الهجوم
    axes[1, 1].plot(t_history, np.degrees(x_history[:-1, 3]))
    axes[1, 1].axhline(y=15, color='r', linestyle='--', label='الحد')
    axes[1, 1].axhline(y=-15, color='r', linestyle='--')
    axes[1, 1].set_ylabel('alpha (deg)')
    axes[1, 1].set_title('زاوية الهجوم')
    axes[1, 1].legend()
    axes[1, 1].grid(True)

    # الارتفاع
    axes[2, 0].plot(t_history, x_history[:-1, 4])
    axes[2, 0].axhline(y=100, color='r', linestyle='--', label='المرجع')
    axes[2, 0].set_ylabel('h (m)')
    axes[2, 0].set_title('الارتفاع')
    axes[2, 0].legend()
    axes[2, 0].grid(True)

    # أمر الزعنفة
    axes[2, 1].plot(t_history, np.degrees(u_history[:, 0]))
    axes[2, 1].axhline(y=25, color='r', linestyle='--', label='الحد')
    axes[2, 1].axhline(y=-25, color='r', linestyle='--')
    axes[2, 1].set_ylabel('delta_e (deg)')
    axes[2, 1].set_title('انحراف الزعنفة')
    axes[2, 1].legend()
    axes[2, 1].grid(True)

    # زمن الحل
    axes[3, 0].plot(t_history, solve_time_history)
    axes[3, 0].axhline(y=10, color='r', linestyle='--', label='ميزانية 10ms')
    axes[3, 0].set_ylabel('زمن الحل (ms)')
    axes[3, 0].set_xlabel('الزمن (s)')
    axes[3, 0].set_title('أداء الحل')
    axes[3, 0].legend()
    axes[3, 0].grid(True)

    # إحصائيات
    axes[3, 1].axis('off')
    stats_text = (
        f"متوسط زمن الحل: {np.mean(solve_time_history):.2f} ms\n"
        f"أقصى زمن حل: {np.max(solve_time_history):.2f} ms\n"
        f"زاوية هجوم قصوى: {np.max(np.abs(np.degrees(x_history[:, 3]))):.1f} deg\n"
        f"انحراف زعنفة أقصى: {np.max(np.abs(np.degrees(u_history))):.1f} deg\n"
    )
    axes[3, 1].text(0.1, 0.5, stats_text, fontsize=12, family='monospace',
                     transform=axes[3, 1].transAxes, verticalalignment='center')

    plt.tight_layout()
    plt.savefig('results/mpc_standalone_test.png', dpi=150)
    plt.show()
    print("تم حفظ النتائج في results/mpc_standalone_test.png")

if __name__ == '__main__':
    simulate_mpc()
```

### 9.5 الخطوة 4: دمج MPC مع محاكاة 6DOF الموجودة

بعد نجاح الاختبار المستقل، ندمج MPC مع المحاكاة الكاملة. أنشئ ملف `6DOF_v2_130/mpc/m130_mpc_autopilot.py`:

```python
"""
طيار آلي MPC متعدد المراحل لصاروخ M130
يعمل كبديل لـ M130AutopilotSystem في المحاكاة
MPC يتحكم في كل المراحل (إطلاق + صعود + نهائي)
يحتفظ بـ PID+PN كاحتياط عند فشل MPC فقط
"""
import numpy as np
import math
from acados_template import AcadosOcpSolver
from dynamics.autopilot import AutopilotInput, AutopilotOutput, AutopilotMode
from dynamics.m130_autopilot import M130AutopilotSystem

class M130MpcAutopilot:
    def __init__(self, config, solver_json='m130_ocp.json'):
        # تحميل الحل متعدد المراحل
        self.solver = AcadosOcpSolver.create_cython_solver(solver_json)
        self.N = 20  # 5+10+5 خطوات

        # الطيار التقليدي كاحتياط فقط
        self.fallback_ap = M130AutopilotSystem(config)

        # حالات داخلية
        self.mpc_active = True  # MPC يعمل من البداية
        self.mpc_fail_count = 0
        self.max_mpc_failures = 5
        self.last_delta_e = 0.0  # آخر أمر تحكم (لقيود معدل التغيّر)

        self.config = config

    def compute(self, autopilot_input: AutopilotInput) -> AutopilotOutput:
        # MPC متعدد المراحل يعمل في كل المراحل
        # الأوزان والقيود مدمجة في الحل (لا حاجة لتبديل يدوي)
        try:
            return self._mpc_compute(autopilot_input)
        except Exception as e:
            self.mpc_fail_count += 1
            if self.mpc_fail_count > self.max_mpc_failures:
                print(f"MPC فشل {self.mpc_fail_count} مرات. الانتقال لـ PID+PN.")
            return self.fallback_ap.compute(autopilot_input)

    def _mpc_compute(self, inp: AutopilotInput) -> AutopilotOutput:
        # استخراج الحالة الحالية من مدخلات الطيار الآلي
        V = inp.airspeed
        gamma = math.atan2(-inp.velocity_ned[2], 
                math.sqrt(inp.velocity_ned[0]**2 + inp.velocity_ned[1]**2))
        q_rate = inp.angular_velocity[1]
        alpha = inp.alpha if hasattr(inp, 'alpha') else 0.0
        h = inp.altitude
        x_pos = 0.0  # لا نتتبع الموقع الأفقي في MPC

        # 7 حالات: 6 أصلية + آخر أمر تحكم (لقيود معدل التغيّر)
        x_current = np.array([V, gamma, q_rate, alpha, h, x_pos, self.last_delta_e])

        # تحديث المعاملات (كتلة، دفع، كثافة، عزم)
        rho = inp.rho if hasattr(inp, 'rho') else 1.225
        params = np.array([inp.mass, inp.thrust, rho, inp.Iyy])

        # تحديث المسار المرجعي (8 = 7 حالات + 1 تحكم)
        gamma_ref = math.atan2(inp.pitch_accel_cmd, 9.81) if inp.pitch_accel_cmd else gamma
        y_ref = np.array([V, gamma_ref, 0.0, 0.0, inp.target_alt, 0.0, 0.0, 0.0])
        y_ref_e = np.array([V, gamma_ref, 0.0, 0.0, inp.target_alt, 0.0, 0.0])

        for k in range(self.N):
            self.solver.set(k, "yref", y_ref)
            self.solver.set(k, "p", params)
        self.solver.set(self.N, "yref", y_ref_e)
        self.solver.set(self.N, "p", params)

        # تعيين الحالة الابتدائية
        self.solver.set(0, "lbx", x_current)
        self.solver.set(0, "ubx", x_current)

        # حل
        status = self.solver.solve()

        if status != 0:
            raise RuntimeError(f"MPC solver failed with status {status}")

        # استخراج أمر التحكم الأمثل
        u_opt = self.solver.get(0, "u")
        delta_pitch = u_opt[0]

        # حفظ آخر أمر تحكم (لقيود معدل التغيّر في الدورة التالية)
        self.last_delta_e = delta_pitch

        # إعادة ضبط عداد الفشل
        self.mpc_fail_count = 0

        # إرجاع الأوامر
        # ملاحظة: MPC يتحكم بالميل فقط حاليًا
        # الانعراج والدوران يستخدمان النظام التقليدي
        fallback = self.fallback_ap.compute(inp)
        return AutopilotOutput(
            delta_roll=fallback.delta_roll,    # دوران: PID التقليدي
            delta_pitch=delta_pitch,           # ميل: MPC
            delta_yaw=fallback.delta_yaw,      # انعراج: PID التقليدي
        )
```

```cpp
// تهيئة MPC (تُستدعى مرة واحدة عند التشغيل)
// ════════════════════════════════════════════════
bool RocketGNC::_mpcInit()
{

    int status = m130_acados_create(_acados_capsule);
    if (status != 0) {
        PX4_ERR("MPC: فشل إنشاء الحل (status=%d)", status);
        return false;
    }

    _mpc_initialized = true;
    PX4_INFO("MPC: تم التهيئة بنجاح");
    return true;
}

// ════════════════════════════════════════════════
// تنظيف MPC (عند الإيقاف)
// ════════════════════════════════════════════════
void RocketGNC::_mpcCleanup()
{
    if (_acados_capsule) {
        m130_acados_free(_acados_capsule);
        m130_acados_free_capsule(_acados_capsule);
        _acados_capsule = nullptr;
    }
    _mpc_initialized = false;
}

// ════════════════════════════════════════════════
// تحديد مرحلة الطيران الحالية
// ════════════════════════════════════════════════
RocketGNC::FlightPhase RocketGNC::_detectPhase(float t, float thrust)
{
    if (thrust > 10.0f) {
        return PHASE_LAUNCH;     // المحرك يعمل
    } else if (t < _t_terminal) {
        return PHASE_CRUISE;     // طيران حر (بعد انتهاء الدفع، قبل المرحلة النهائية)
    } else {
        return PHASE_TERMINAL;   // توجيه نهائي (آخر 2-3 ثوانٍ)
    }
}

// ════════════════════════════════════════════════
// حل MPC متعدد المراحل (تُستدعى كل 10ms)
// ════════════════════════════════════════════════
// الأوزان والقيود مُدمجة في الحل (خطوات 0-4: إطلاق، 5-14: صعود، 15-19: نهائي)
// هنا نحدّث فقط: الحالة الحالية + المعاملات + المسار المرجعي
float RocketGNC::_mpcSolve(
    float V, float gamma, float q_rate, float alpha, float h,
    float last_delta_e,
    float mass, float thrust, float rho, float Iyy,
    float gamma_ref, float h_ref)
{
    if (!_mpc_initialized || !_mpc_enabled) {
        return NAN;  // إشارة للاستخدام PN
    }

    ocp_nlp_config *nlp_config = m130_acados_get_nlp_config(_acados_capsule);
    ocp_nlp_dims *nlp_dims = m130_acados_get_nlp_dims(_acados_capsule);
    ocp_nlp_in *nlp_in = m130_acados_get_nlp_in(_acados_capsule);
    ocp_nlp_out *nlp_out = m130_acados_get_nlp_out(_acados_capsule);

    // 1. تعيين الحالة الحالية (7 حالات: 6 أصلية + delta_e_prev)
    double x0[7] = {V, gamma, q_rate, alpha, h, 0.0, last_delta_e};
    ocp_nlp_constraints_model_set(nlp_config, nlp_dims, nlp_in, 0, "lbx", x0);
    ocp_nlp_constraints_model_set(nlp_config, nlp_dims, nlp_in, 0, "ubx", x0);

    // 2. تحديث المعاملات
    double p[4] = {mass, thrust, rho, Iyy};
    for (int k = 0; k <= 20; k++) {
        m130_acados_update_params(_acados_capsule, k, p, 4);
    }

    // 3. تحديث المسار المرجعي (8 = 7 حالات + 1 تحكم)
    double yref[8] = {V, gamma_ref, 0.0, 0.0, h_ref, 0.0, 0.0, 0.0};
    double yref_e[7] = {V, gamma_ref, 0.0, 0.0, h_ref, 0.0, 0.0};
    for (int k = 0; k < 20; k++) {
        ocp_nlp_cost_model_set(nlp_config, nlp_dims, nlp_in, k, "yref", yref);
    }
    ocp_nlp_cost_model_set(nlp_config, nlp_dims, nlp_in, 20, "yref", yref_e);

    // 4. حل المسألة
    hrt_abstime t_start = hrt_absolute_time();
    int status = m130_acados_solve(_acados_capsule);
    float solve_time_ms = (hrt_absolute_time() - t_start) / 1000.0f;

    // 5. فحص النتيجة
    if (status != 0) {
        _mpc_fail_count++;
        if (_mpc_fail_count > MPC_MAX_FAILURES) {
            PX4_WARN("MPC: %d فشل متتالي، إيقاف MPC", _mpc_fail_count);
            _mpc_enabled = false;
        }
        return NAN;
    }

    _mpc_fail_count = 0;

    // 6. استخراج أمر التحكم الأمثل
    double u_opt[1];
    ocp_nlp_out_get(nlp_config, nlp_dims, nlp_out, 0, "u", u_opt);

    if (solve_time_ms > 8.0f) {
        PX4_WARN("MPC: زمن الحل طويل: %.1f ms", (double)solve_time_ms);
    }

    return (float)u_opt[0];
}
```

```
             └── مختلف  → تحقق وعدّل
```

### 10.10 نصائح مهمة للنقل

**1. لا تعدّل الكود المولَّد يدويًا:**
إذا أردت تغيير أي شيء (أوزان، قيود، نموذج)، عدّل ملف Python وأعد التوليد. الكود المولَّد يجب أن يبقى كما هو.

**2. ابدأ بسيطًا:**
- ابدأ بنموذج 2D (مستوى الميل فقط) ← اختبره
- ثم وسّع لـ 3D (ميل + انعراج) ← اختبره
- ثم أضف دوران ← اختبره

**3. احتفظ دائمًا بالاحتياط:**
لا تحذف كود PID+PN أبدًا. MPC يجب أن يكون إضافة وليس استبدالاً. معامل `ROCKET_MPC_EN` يسمح بالتبديل في أي وقت.

**4. راقب زمن الحل:**
إذا كان زمن الحل أكثر من 5ms على الهاتف:
- قلل `N` من 20 إلى 10 (خطوات أقل = حل أسرع)
- قلل `sim_method_num_stages` من 4 إلى 2 (RK2 بدلاً من RK4)
- بسّط النموذج (احذف المتغيرات غير الضرورية)

**5. دورة التطوير:**
أي تعديل في النموذج أو الأوزان:
```
عدّل Python → أعد توليد C → انسخ لـ PX4 → ابنِ → اختبر HITL
```
هذه الدورة يجب أن تكون سلسة وسريعة. لا تعدّل كود C يدويًا.

## 11. ما الذي سيعالجه MPC ولم يكن PID يعالجه؟

### 11.1 مشاكل كان PID يعاني منها ولم تكن تعرف السبب

```
المشكلة 1: تشبّع الزعنفة + تراكم التكامل (Windup)
═══════════════════════════════════════════════════════

PID:
  الخطأ كبير → PID يطلب δ=40° → الزعنفة حدها 25°
  → يبقى عند 25° → التكامل (Ki) يتراكم في الخلفية
  → عندما يقل الخطأ، الاستجابة تتأخر بسبب التراكم
  → تأرجح غير متوقع (overshoot)

  هل كنت تتوقع هذا؟ غالبًا لا. كنت ترى تأرجحًا ولا تعرف أنه بسبب windup.

MPC:
  يعرف أن الحد 25° من البداية
  → لا يطلب 40° أبدًا → لا يوجد windup لأنه لا يوجد تكامل
  ✓ المشكلة لا تحدث أصلاً
```

```
المشكلة 2: التداخل بين المحاور (Cross-Coupling)
══════════════════════════════════════════════════

PID:
  3 حلقات مستقلة (pitch, yaw, roll)
  حلقة الميل تعطي أمرًا → يسبب تأثيرًا على الانعراج
  حلقة الانعراج تصحح → يسبب تأثيرًا على الميل
  → تذبذب متبادل لم تكن تتوقعه

  هل كنت تتوقع هذا؟ لا. كنت ترى اهتزازًا غريبًا ولا تعرف مصدره.

MPC:
  يحل pitch + yaw معًا كمسألة واحدة
  → يعرف أن أمر الميل سيؤثر على الانعراج
  → يعوّض التأثير مسبقًا في نفس الحل
  ✓ المشكلة لا تحدث أصلاً
```

```
المشكلة 3: تغير ديناميكا الصاروخ أثناء الطيران
══════════════════════════════════════════════════

PID:
  مكاسب ثابتة أو مجدولة (gain scheduling)
  لكن الجدولة تعتمد على q_dyn فقط
  ماذا لو تغيرت الكتلة (احتراق الوقود) + تغير مركز الثقل
  + تغير Mach كلها في نفس الوقت؟
  → الجدولة لا تغطي كل التوليفات

  هل كنت تتوقع هذا؟ لا. ضبطت المكاسب عند ظروف معينة
  وتأمل أنها تعمل دائمًا.

MPC:
  يستقبل الكتلة والدفع والكثافة كمعاملات كل 10ms
  → يعيد حساب الأمر الأمثل للظروف الحالية بالضبط
  → لا يحتاج جدولة لأنه يحل المسألة من الصفر كل مرة
  ✓ يتكيف تلقائيًا
```

```
المشكلة 4: اقتراب من حدود زاوية الهجوم
═══════════════════════════════════════

PID:
  لا يعرف أن α=14° خطير (الحد 15°)
  يعطي أمرًا يزيد α إلى 18° → فقدان الاستقرار الايروديناميكي

  هل كنت تتوقع هذا؟ ربما لا. يحدث فقط في مناورات حادة.

MPC:
  يعرف أن |α| ≤ 15° قيد صلب
  عندما α=12°: يبدأ بتخفيف المناورة تلقائيًا
  ✓ حماية مدمجة بدون كود إضافي
```

### 11.2 مشاكل كنت تحلها يدويًا بالتجربة والخطأ

```
ضبط المكاسب (Tuning):
─────────────────────
PID: تضبط Kp, Ki, Kd بالتجربة. تغيّر واحدًا فيتأثر الآخران.
     أسابيع من الضبط ولا تعرف إن كان الأمثل أم لا.

MPC: تحدد "ماذا أريد" (الأوزان) بدلاً من "كيف أفعل" (المكاسب).
     W_gamma=100 يعني "الاتجاه مهم". لا تحتاج حساب Kp/Ki/Kd.
```

```
الاستجابة البطيئة مقابل التأرجح:
─────────────────────────────────
PID: Kp كبير = استجابة سريعة لكن تأرجح
     Kp صغير = لا تأرجح لكن بطيء
     تبحث عن حل وسط ولا تجده دائمًا.

MPC: يتنبأ بالمستقبل. يعرف أنه لو أعطى أمرًا قويًا الآن سيتأرجح.
     فيعطي أمرًا متوسطًا الآن ويكمله في الخطوة التالية.
     → استجابة سريعة بدون تأرجح (لأنه يخطط للأمام).
```

```
تعويض انحراف الدفع (Thrust Misalignment):
──────────────────────────────────────────
PID: اكتشفت المشكلة وعملت عليها في run_misalignment_test.py
     التعويض يتم محور بمحور. pitch يعوّض ثم yaw يعوّض.
     لكن التعويضين يتعارضان أحيانًا.

MPC: يُدخل انحراف الدفع كمعامل في النموذج.
     يحسب التعويض الأمثل عبر كل المحاور معًا في حل واحد.
     → تعويض متكامل بدون تعارض.
```

### 11.3 جدول ملخص شامل

| المشكلة | هل كنت تتوقعها مع PID؟ | هل MPC يعالجها؟ | كيف؟ |
|---------|------------------------|----------------|------|
| تشبّع الزعنفة + windup | غالبًا لا | نعم | القيود مدمجة، لا يوجد تكامل |
| تداخل pitch-yaw | غالبًا لا | نعم | يحل المحاور معًا |
| تغير الكتلة أثناء الاحتراق | تعرفها لكن صعب التعويض | نعم | يستقبل الكتلة كمعامل كل 10ms |
| تجاوز زاوية الهجوم | في المناورات الحادة فقط | نعم | قيد صلب \|α\|≤15° |
| تأرجح بعد مناورة | نعم لكن صعب حله | نعم | يتنبأ ويمنعه مسبقًا |
| انحراف محور الدفع | اكتشفتها وعملت عليها | نعم، أفضل | يعوّض عبر المحاور معًا |
| اختلاف الأداء بين السرعات | نعم (gain scheduling) | نعم، تلقائيًا | يعيد الحل كل 10ms |
| أوامر متعارضة بين المحاور | غالبًا لا | نعم | حل واحد متكامل |
| تجاوز حدود حركة الزعنفة | غالبًا لا | نعم | قيد على سرعة الزعنفة |

### 11.4 لكن: ما لا يعالجه MPC (بصراحة)

| المشكلة | لماذا MPC لا يعالجها | من يعالجها |
|---------|---------------------|-----------|
| عطل المستشعر (GPS انقطع، IMU خطأ) | يعتمد على بيانات صحيحة | EKF2 (كشف الخطأ + دمج المستشعرات) |
| ظاهرة فيزيائية غير موجودة في النموذج (flutter) | لا يعرفها | يجب إضافتها للنموذج أو الاحتياط (PN) |
| عطل ميكانيكي (زعنفة مكسورة) | لا يمكن حله بالرياضيات وحدها | تصميم ميكانيكي متين |
| تأخير كبير في الـ servo (>50ms) | أوامره تصل متأخرة | استخدام servo أسرع |
| اضطراب أكبر من قدرة الصاروخ | لا أحد يستطيع | تصميم ايروديناميكي أفضل |

### 11.5 الفرق الجوهري

```
PID:  "حدث خطأ → أرى الخطأ → أصحح → أتمنى ألا يتأرجح"
       ↑ يتفاعل مع الماضي فقط
       ↑ لا يعرف القيود
       ↑ كل محور وحده

MPC:  "أرى الوضع الآن → أحسب أفضل خطة للثانيتين القادمتين
       مع ضمان عدم تجاوز أي حد → أنفذ الخطوة الأولى فقط
       → بعد 10ms أعيد الحساب بمعلومات جديدة"
       ↑ يخطط للمستقبل
       ↑ يضمن القيود رياضيًا
       ↑ كل المحاور معًا
```

**أغلب المشاكل "غير المتوقعة" مع PID ليست غير متوقعة فعلاً - هي نتائج طبيعية لمحدودية PID (لا يعرف القيود، لا يتنبأ، لا يربط المحاور). MPC يزيل هذه المحدوديات من الأساس.**

### 12.4 EKF2: لا يعرف أن MPC موجود

```

  EKF2 → vehicle_attitude, vehicle_local_position
  (لا يعرف من يقرأ بياناته)

  EKF2 ← sensor_combined     ← نفس المدخلات
  EKF2 → vehicle_attitude     ← نفس المخرجات
  (لا يزال لا يعرف من يقرأ بياناته)

  EKF2 لا يعرف ولا يهتم إن كان RocketGNC يستخدم PID أو MPC.
  هو فقط يقدّر الحالة وينشرها.
```

## 13. هل acados سيعمل فعلاً؟ الأدلة من الواقع

### الدليل الأقوى: ما فعلناه بـ PX4 أصعب مما سنفعله بـ acados

```
ما فعلناه مع PX4 (نجح):                    ما سنفعله مع acados:
════════════════════════                     ═══════════════════════

PX4 مصمم لـ:                                acados مصمم لـ:
- طائرات بأربعة محركات                       - التحكم الأمثل في الزمن الحقيقي
- إقلاع وهبوط عمودي                          - أنظمة مدمجة
- طيران بطيء ومستقر                          - حل مسائل OCP بسرعة

ما فعلناه:                                   ما سنفعله:
- حوّلناه لصاروخ بأربع زعانف                 - نضيف وحدة MPC بجانب الوحدات الموجودة
- شغّلناه على هاتف أندرويد (ليس FCU)         - نربطها بـ uORB الموجود أصلاً
- كتبنا وحدة RocketGNC من الصفر              - نستخدم كود C مولَّد جاهز
- ربطنا المستشعرات عبر NDK                    - نربط مكتبة C عبر CMake
- بنينا جسور MAVLink مخصصة                   - نستدعي دالة واحدة كل 10ms
- حللنا مشاكل uORB على أندرويد
- صنعنا نظام HITL كامل

الصعوبة: ██████████ (10/10)                  الصعوبة: ████ (4/10)
```

**إذا استطعت تحويل PX4 من نظام كوادروتر إلى نظام صاروخي على هاتف أندرويد، فإضافة مكتبة C واستدعاء دالة منها أبسط بكثير.**

### أدلة من مشاريع حقيقية تعمل الآن

#### 1. openpilot (comma.ai) - آلاف السيارات يوميًا

```
النظام: قيادة ذاتية حقيقية على الطرقات
المعالج: Snapdragon 845 (نفس عائلة معالجك)
الحل: acados/ACADO MPC للتحكم الجانبي
التردد: 20Hz
الحالة: يعمل على آلاف السيارات كل يوم منذ سنوات
```

هذا دليل قاطع: acados يعمل على Snapdragon في الزمن الحقيقي، في نظام أخطاؤه تقتل بشرًا.

#### 2. طائرات بدون طيار (بحث + تطبيق)

```
النظام: NMPC للتحكم بالموقع والسرعة
المعالج: Jetson Xavier NX (ARM مثل Snapdragon)
زمن الحل: 4ms على حاسوب، 7.3ms على ARM
التردد: 100Hz (نفس ما نحتاجه)
```

#### 3. أنظمة فضائية

```
النظام: إدارة مشغلات مركبة فضائية
الاختبار: Processor-in-the-Loop على عتاد فضائي حقيقي
النتيجة: يعمل بكفاءة عالية
```

#### 4. متحكمات دقيقة (أضعف 100x من هاتفك)

```
STM32H7 (Cortex-M7, 246MHz): زمن حل 182ms (N=20)
نفس المسألة على هاتفك (~2.8GHz, arm64): متوقع 4-9ms
```

### لماذا سينجح تحديدًا مع M130

| السبب | التفصيل |
|-------|---------|
| **البنية جاهزة** | uORB، CMake، JNI، NDK - كل البنية التحتية موجودة من عمل PX4 |
| **مسألة بسيطة** | 7 حالات (6 ديناميكية + تتبع التحكم)، 1 تحكم، N=20 - هذه مسألة صغيرة لـ acados |
| **مدة قصيرة** | الصاروخ يطير 10-30 ثانية - لا مشاكل حرارية أو استقرار طويل المدى |
| **احتياط موجود** | PN+PID يعملان ومُختبران - MPC إضافة وليس استبدال |
| **SQP_RTI** | تكرار واحد في كل دورة - أسرع بكثير من SQP الكامل |
| **BLASFEO محسّن لـ ARM** | المكتبة مُحسَّنة خصيصًا لمعمارية ARMV8A |

### المقارنة الحاسمة

```
openpilot (يعمل الآن):              M130 (ما سنفعله):
══════════════════════              ═════════════════════

معالج: Snapdragon 845              معالج: Snapdragon (مماثل أو أحدث)
تردد الحل: 20Hz                    تردد الحل: 100Hz (أعلى)
عدد الحالات: ~6                     عدد الحالات: 6 (مماثل)
عدد التحكم: 2                       عدد التحكم: 1 (أبسط)
مدة التشغيل: ساعات                  مدة التشغيل: ثوانٍ (أسهل حراريًا)
عواقب الخطأ: حادث سيارة             عواقب الخطأ: fallback لـ PN
النتيجة: يعمل بنجاح ✓              النتيجة: سيعمل ✓
```

### الخلاصة

تحويل PX4 من نظام كوادروتر إلى نظام صاروخي على هاتف أندرويد كان إنجازًا هندسيًا كبيرًا تطلب:
- كتابة وحدة GNC كاملة من الصفر
- حل مشاكل توافق uORB مع أندرويد
- بناء جسور مستشعرات عبر NDK
- بناء جسور اتصال MAVLink مخصصة
- إنشاء نظام HITL كامل

إضافة acados بالمقارنة هي:
- ربط مكتبة C عبر CMake (موجود أصلاً)
- إضافة ملفات C مولَّدة لوحدة RocketGNC
- استدعاء دالة واحدة (`m130_acados_solve`) كل 10ms
- إضافة 3 معاملات PX4

**إذا نجح الأصعب، سينجح الأسهل.**

## ملاحظة أخيرة

acados ليس بديلاً للنظام الحالي بل **ترقية ذكية**. النظام الحالي (PID + PN) يبقى كطبقة أمان احتياطية، وMPC يضاف كطبقة علوية تعمل عندما تكون الظروف مناسبة. هذا النهج المتدرج يضمن أن الصاروخ لن يفقد السيطرة أبدًا حتى لو فشل MPC.
